# 待完善清单

本文基于当前仓库代码现状整理“比较明显的半成品 / 风险点”，重点覆盖移动端 App、Go 后端、Web 管理端三部分。目标不是罗列所有细节，而是优先标出会影响安全、数据正确性、联调闭环和后续扩展的事项，便于排期。

## 总体判断

项目已经具备完整产品骨架：

- Flutter 客户端已经有登录注册、首页 Tab、记录增删改查、基础同步能力。
- Go 后端已经有认证、时光、上传、管理端接口、用户 / 角色 / 权限 / 日志模型。
- Vue 管理端已经有登录、主布局、用户 / 时光 / 角色 / 权限 / 日志页面。

当前最明显的问题不在“有没有页面”，而在以下四类：

- 安全边界还没有完全闭环。
- 本地数据与云端数据模型还没有完全统一。
- 媒体相关链路只做了一半。
- 自动化测试和变更兜底能力明显不足。

## 优先级建议

| 优先级 | 建议处理范围 | 目标 |
| --- | --- | --- |
| P0 | 立即处理 | 先消除安全和数据串号风险，打通最基本的线上可用闭环 |
| P1 | 尽快处理 | 提升同步、体验和维护性，避免后续返工 |
| P2 | 规划处理 | 补齐工程化和容量演进能力 |

## P0

### 1. 本地 SQLite 数据未按账号隔离

**现状**

- 本地表 `moments` 只有记录字段和 `synced` 标记，没有 `user_id`。
- 退出登录只清理 `SharedPreferences` 中的 token 和用户信息，不清理本地 SQLite。
- `MomentProvider.initialize()` 会直接加载全部本地记录。

**影响**

- 切换账号后，后一个账号可能直接看到前一个账号的本地离线记录。
- 这属于明显的数据隔离问题，优先级应视为高。
- 后续如果继续做“离线可编辑 + 登录后同步”，当前结构很容易产生串号和误同步。

**建议动作**

- 给本地表增加 `user_id` 或 `owner_scope` 字段。
- 登录后仅加载当前用户对应的本地记录。
- 退出登录时明确策略：
  - 要么保留多账号本地数据，但必须按账号隔离。
  - 要么仅保留当前账号数据，切换账号时清空本地表。
- 同步逻辑也要改为“只同步当前账号的未同步记录”。

**验收标准**

- A 账号离线创建的数据，B 账号登录后不可见。
- A 账号退出再登录，仍能看到自己的离线数据。
- 同步时不会把 A 账号本地数据上传到 B 账号云端。

---

### 2. 用户侧时光接口存在未鉴权 / 越权读取风险

**现状**

- `GET /v1/moments` 路由未挂认证中间件。
- 该接口内部通过 `GetUserID(c)` 取用户 ID；未登录时会得到 `0`。
- 仓储层 `FindAll` 在 `user_id > 0` 时才追加过滤条件，因此未登录访问时可能读取全量时光数据。
- `GET /v1/moments/:id` 同样未做认证和归属校验，只要知道记录 ID 就能取详情。

**影响**

- 这是直接的数据泄漏风险，不是体验问题。
- 只要接口对外暴露，就存在被遍历和抓取的可能。
- 一旦上线，风险等级高于大部分功能缺失。

**建议动作**

- 为用户侧 `GET /v1/moments`、`GET /v1/moments/:id` 增加认证中间件。
- 在服务层或仓储层增加归属校验，保证用户只能读取自己的记录。
- 如果后续确实需要“公开时光”能力，应单独设计公开接口和公开字段，不要复用私有接口。

**验收标准**

- 未登录访问用户时光接口返回未授权。
- 普通用户不能读取其他用户的时光列表或详情。
- 相关接口有最基本的鉴权测试覆盖。

---

### 3. 管理端接口只有登录分流，没有真正的管理员鉴权边界

**现状**

- 管理员登录会签发 `role=admin` 的 token。
- 但 `/v1/admin/**` 的保护主要依赖通用 `Auth()` 中间件。
- 当前未看到专门的 `RequireAdmin` 或权限码校验中间件。
- 这意味着只要拿到一个有效用户 token，就可能直接请求管理端 API。

**影响**

- 管理端“有角色表 / 权限表”不等于“真正做了授权控制”。
- 当前更接近“管理端有后台页面，但接口授权边界没完全落地”。
- 这是明显的安全半成品。

**建议动作**

- 增加管理员专用中间件，例如：
  - 校验 token 中的 `role` 必须为 `admin`。
  - 或者到数据库校验用户是否具备 `admin/super_admin` 角色。
- 再往下一步，按接口挂权限码，而不是只区分 admin / non-admin。
- 前端菜单和按钮显隐可以后补，但后端接口授权必须先补。

**验收标准**

- 普通用户 token 请求 `/v1/admin/**` 返回无权限。
- 管理员 token 可以访问管理端接口。
- 关键接口至少具备角色级测试。

---

### 4. 媒体“上传到云端并可访问”的主链路没有打通

**现状**

- App 添加图片 / 视频 / 录音时，当前保存的是本地文件路径。
- App 端没有真正调用 `/v1/upload` 上传文件。
- 服务端虽然有上传接口和本地落盘逻辑，但未看到静态文件访问路由或统一访问域名映射。
- 管理端详情页对媒体路径的处理也更像“展示字符串”，不是完整预览。

**影响**

- 现在的“同步到云端”更像是在同步记录元数据，不是同步媒体资产。
- 本地路径上传到服务端后，对其他设备、管理端、后续重装后的客户端都没有可用价值。
- 这会直接影响“图 / 音 / 视频时光”的产品闭环。

**建议动作**

- App 侧新增媒体上传流程：
  - 先上传文件，拿到可访问 URL。
  - 再调用创建时光接口提交云端地址。
- 服务端补齐静态资源访问方案：
  - 本地存储时提供 `/uploads/...` 可访问路由，或返回完整 CDN / OSS URL。
- 管理端和客户端详情页统一按可访问 URL 渲染。
- 明确媒体删除策略：
  - 删除记录时是否同步删除远端媒体。
  - 是否做引用计数或延迟清理。

**验收标准**

- 登录用户创建带媒体的时光后，管理端可以查看媒体。
- 新设备登录同一账号后，可以读取并展示历史媒体。
- 服务端返回的媒体路径可以直接访问。

---

### 5. 本地模型与服务端模型未统一，同步策略仍是半成品

**现状**

- 本地支持 `mixed` 媒体类型，服务端只接受 `text/image/audio/video`。
- 客户端把 `mixed` 上传时降级成 `text`。
- 本地记录主键是 UUID，服务端记录主键是自增数字，当前通过“同步后替换本地行”硬切。
- `MomentProvider` 中明确留有同步冲突处理 TODO。
- “重置同步标记后重新上传”会有重复数据风险，界面文案也已经在提示这个问题。

**影响**

- 同一条记录在本地和云端的语义不完全一致。
- 混合媒体、重复同步、冲突合并这些边界场景现在都不稳定。
- 继续迭代编辑、跨设备同步、多端并发修改时，返工成本会很高。

**建议动作**

- 先统一数据模型，至少明确：
  - 服务端是否支持 `mixed`。
  - `media_paths` 里存的是本地路径、相对路径还是绝对 URL。
  - 客户端本地主键与服务端主键如何映射。
- 增加同步状态字段，例如：
  - `local_id`
  - `server_id`
  - `sync_status`
  - `last_synced_at`
- 明确冲突策略：
  - 本地优先
  - 服务端优先
  - 按 `updated_at` 决定
  - 或进入待确认队列
- 增加幂等机制，避免“重复上传生成多条云端数据”。

**验收标准**

- 一条记录重复同步不会在服务端生成重复数据。
- 混合媒体记录在本地、服务端、管理端语义一致。
- 双端修改同一条记录时有明确、可预期的结果。

## P1

### 6. App 端 refresh token 流程没有闭环

**现状**

- 存储层已经有 `refresh_token` 读写接口。
- 认证接口也会返回 `refresh_token`。
- 但 App 端当前只保存 `access_token`，没有真正存储和使用 `refresh_token`。
- 遇到 `401` 时直接清理登录态，不尝试自动刷新。

**影响**

- 用户登录态体验偏脆弱。
- 一旦 access token 过期，App 会直接掉回登录页。
- 现有代码结构已经在为 refresh token 做准备，但业务层没接完。

**建议动作**

- 登录 / 注册成功后持久化 `refresh_token`。
- 在 `ApiService` 中增加单次刷新和请求重放逻辑。
- 刷新失败时再统一退出登录。

**验收标准**

- access token 过期时，普通请求可自动刷新并继续成功。
- refresh token 失效时，才要求重新登录。

---

### 7. “从服务端拉取并合并”能力存在，但没有真正接进用户流程

**现状**

- `MomentProvider.fetchFromServer()` 已实现基础拉取和合并。
- 当前未看到它被登录成功、首页初始化、手动操作入口等流程实际调用。
- 当前 App 更接近“本地优先 + 手动上传”，不是“云端双向同步”。

**影响**

- 同一账号跨设备数据不能稳定汇聚。
- 用户容易以为自己登录后已经云端同步，实际只做了本地读写。
- 后续数据问题会很难排查，因为“代码里有同步能力”但用户路径没有真正触发。

**建议动作**

- 在登录成功后、首页首次进入时、手动下拉刷新时，明确接入拉取逻辑。
- 给 UI 明确展示“最近同步时间 / 同步状态 / 失败原因”。
- 区分“本地记录”“已同步记录”“拉取失败记录”。

**验收标准**

- 新设备登录同账号后，能主动拉到云端数据。
- 用户可以在 UI 中理解当前同步状态。

---

### 8. 本地数据库版本和迁移机制太薄，后续改表会很痛

**现状**

- SQLite 当前版本是 `1`。
- 只有 `onCreate`，没有 `onUpgrade` / `onDowngrade`。
- 一旦补 `user_id`、`server_id`、`sync_status` 等字段，现有安装数据就会迁移困难。

**影响**

- 当前越晚补本地数据结构，代价越大。
- 如果后续直接改表定义但不做迁移，老用户数据很容易损坏或丢失。

**建议动作**

- 尽快设计 SQLite 迁移策略。
- 补齐版本升级路径和必要的数据回填逻辑。
- 把“账号隔离”和“同步映射”需要的字段一次补齐，避免反复改 schema。

**验收标准**

- 老版本 App 升级后，本地历史记录可保留。
- 新增字段能被正确回填和使用。

---

### 9. 测试覆盖明显不足

**现状**

- Flutter 侧只有一个启动级别 widget test。
- 当前未看到 Go 后端接口测试、服务层测试、权限测试。
- 当前未看到管理端单测或 E2E。

**影响**

- 同步、权限、账号切换这类高风险功能改动时几乎没有自动兜底。
- 现在项目已经跨三端协作，没有测试会直接放大返工成本。

**建议动作**

- P0 改动对应补最小必要测试：
  - 后端鉴权测试
  - 同步逻辑测试
  - 本地账号隔离测试
- 后续补一层最小 smoke 测试，覆盖登录、创建时光、管理端查询。

**验收标准**

- 安全边界和同步核心路径至少有自动测试。
- 关键功能改动后可在 CI 中快速发现回归。

---

### 10. 文档与代码实现已经出现偏差

**现状**

- 根 README 仍偏早期离线版描述。
- 实际项目已经发展为“Flutter + Go + Admin”全栈仓库。
- 文档中一些说明仍以“可选后端”或早期结构为主，容易误导新接手的人。

**影响**

- 新成员容易按旧文档理解项目边界。
- 联调、排障、部署说明会出现“文档说能用，实际代码不是这样”的情况。

**建议动作**

- README 按当前现状重写：
  - 项目组成
  - 标准启动顺序
  - 当前已完成能力
  - 已知限制
- 将“客户端离线版”与“当前全栈版”区别说明清楚。

**验收标准**

- 新成员只看 README 就能理解仓库组成和正确启动方式。
- 文档不再把当前项目误写成单一 Flutter App。

## P2

### 11. Redis 已在配置和编排中出现，但业务中基本未落地

**现状**

- `docker-compose.yml`、配置文件里都包含 Redis。
- 当前后端代码中未看到明确的 Redis 连接和实际使用场景。

**影响**

- 本地和部署环境引入了额外依赖，但暂时没有真实收益。
- 会增加运维和理解成本。

**建议动作**

- 如果短期不用，文档中明确“当前保留但未启用”。
- 如果要用，优先考虑落地到：
  - 刷新 token 管理
  - 登录态黑名单
  - 验证码 / 限流
  - 热点缓存

**验收标准**

- Redis 要么有清晰用途，要么在启动文档中被降级为可选依赖。

---

### 12. 统一响应永远返回 HTTP 200，客户端已经开始到处做兼容

**现状**

- 后端业务错误大多以 HTTP 200 + `body.code != 200` 返回。
- App 和 Admin 都已经写了额外兼容逻辑。

**影响**

- 与常见 HTTP 语义不一致。
- 网关、监控、重试、浏览器网络面板都会更难判断真错误。
- 客户端每个调用点都要额外做一层解析，长期维护成本高。

**建议动作**

- 新接口逐步回归标准 HTTP 状态码。
- 保留 `code/msg/data` 结构可以，但不要把所有错误都塞进 HTTP 200。
- 至少从认证、权限、参数错误这几类高频场景先改。

**验收标准**

- 常见错误类型能通过 HTTP 状态码直接识别。
- 客户端不再需要在多处为“HTTP 200 但业务失败”写特殊分支。

---

### 13. 管理端权限模型还停留在“数据层有结构，界面层有基础页”

**现状**

- 角色、权限、角色权限关系表已经存在。
- 管理端也有角色和权限页面。
- 但菜单显隐、按钮级禁用、接口级权限码校验还没形成完整闭环。

**影响**

- 当前更像“已经搭出权限系统的数据底座”，不是“权限系统已完工”。
- 若后面新增更多管理动作，没有完整授权体系会越来越乱。

**建议动作**

- 先把后端接口级权限控制补齐。
- 再补前端菜单和按钮级显隐。
- 最后再考虑更细的资源级权限。

**验收标准**

- 菜单、按钮、接口三层权限含义一致。
- 权限不是“仅存表”，而是实际影响可见性和可操作性。

## 建议排期顺序

建议按下面顺序推进：

1. 修复用户侧时光接口未鉴权 / 越权问题。
2. 修复管理端 API 的管理员鉴权边界。
3. 改造本地 SQLite，完成按账号隔离。
4. 统一同步模型，补 `server_id / user_id / sync_status` 等必要字段。
5. 打通媒体上传、存储、访问主链路。
6. 接入 App 端 refresh token 和拉取同步流程。
7. 补最小必要测试。
8. 回写 README 和调试文档。

## 结论

当前项目最值得警惕的不是“页面还少一点”，而是：

- 有几处安全边界实际上没有锁住。
- 离线数据和账号体系还没有真正结合好。
- 媒体能力在产品表面上存在，但云端主链路没闭环。
- 测试和迁移机制不足，会放大后续改动成本。

如果只做表面功能追加，不先处理上述 P0 项，后面会越来越难收拾。
